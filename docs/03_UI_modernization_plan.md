# План модернизации UI: современный фронт поверх Moodle

> Тезис заказчика верный: **движок и логика Moodle — отличные, UI устарел.** Вопрос только в том,
> *насколько глубоко* переписывать фронт — и это развилка с кардинально разной ценой/риском.

## Три стратегии (честно, с ценой)

### A. Глубокий theming (остаёмся в рендере Moodle)
Переопределяем все Mustache-шаблоны + layout'ы + SCSS + немного JS. Тема как Boost Union/RemUI.
- ✅ Сохраняется ВСЁ: функции, админка, моб. приложение, плагины. Быстро (дни-недели).
- ✅ Безопасно для тендера, минимальный maintenance при апгрейдах Moodle.
- ❌ Это всё ещё server-rendered Bootstrap. Можно сделать красиво и современно (~90% вида), но не «SPA-ощущение».

### B. Headless — полностью свой фронт (настоящий «переписать весь UI»)
Moodle = бэкенд/API. Новый фронт на React/Next.js потребляет Moodle Web Services (REST) + наши кастом-эндпоинты.
- ✅ Полная свобода дизайна, современный SPA/PWA, своя компонентная система.
- ❌ **Огромный объём.** Moodle WS API обширный, но НЕ полный: плеер тестов, SCORM, редактирование курсов, отчёты, сертификаты — выставлены частично или болезненно. Это переписывание гигантской поверхности.
- ❌ Теряем готовую админку и официальное моб. приложение; ломаем UI плагинов.
- ❌ Каждый апгрейд ядра может менять WS-ответы → постоянный maintenance.
- ❌ Для тендера до 12.2026 паритета не достичь; растёт security-аудит (вторая система).

### C. Гибрид — современный фронт на путь УЧЕНИКА, Moodle-UI на сложное (РЕКОМЕНДУЮ)
Новый React-фронт для самых частых сценариев ученика (логин, дашборд, каталог, страница курса, «моё обучение»,
профиль, сертификаты). Сложное (попытки тестов, SCORM, авторинг курсов, админка, отчёты) — оставляем в
**глубоко затемённом** Moodle (вариант A), встраиваем/стилизуем.
- ✅ Современный вид там, где его видят 95% пользователей; объём ограничен и управляем.
- ✅ Сложные/рискованные потоки продолжают работать на зрелом Moodle.
- ❌ Два UI и мост сессий/SSO между ними.

## Рекомендация
**Делать A и C параллельно, не делать B как первый шаг.**
1. **Для ТЕНДЕРА** (срок 12.2026): везём **A** — глубоко затемённый Moodle (Boost Union + наш бренд индиго/Lexend). Это tender-ready, проходит ИБ, без рисков по срокам.
2. **Как продукт-дифференциатор**: строим **C** — headless-фронт для журналов ученика. Это и есть «современный UI», но фазами и без ставки на него в дедлайне.
3. **B (полный headless)** — только если после фазы C бизнес решит, что окупается. Не раньше.

> Ставить тендер на переписанный с нуля фронт = почти гарантированно сорвать срок/паритет. Поэтому фронт — стратегический продукт ПОВЕРХ tender-ready базы, а не вместо неё.

## Архитектура варианта C (рекомендуемого)
```
┌────────────────────────┐         ┌──────────────────────────────┐
│  Новый фронт (Next.js)  │  HTTPS  │  Moodle 4.5 (наш дистрибутив) │
│  React + TS + Tailwind  │ ──────► │  • Web Services REST (ядро)   │
│  + shadcn/ui            │  token  │  • local_corplms: свои        │
│  + TanStack Query (PWA) │ ◄────── │    external functions (доп.)  │
│  Дизайн-система UX/UI Pro│         │  • Глубокая тема для встраив. │
└────────────────────────┘         │    сложных потоков (quiz/SCORM)│
        │ встраивает/стилизует ───► └──────────────────────────────┘
        ▼
  Quiz attempt / SCORM / авторинг / админка = themed Moodle (iframe/seamless)
```

**Стек фронта:** Next.js (React, TS), Tailwind + **shadcn/ui**, TanStack Query, next-pwa. Дизайн-токены из
скилла UX/UI Pro (indigo `#4F46E5`, success `#22C55E`, Lexend/Source Sans 3, «Trust & Authority»).
**Аутентификация:** Moodle как identity — SSO/OAuth2, per-user WS-токен. Сессионный мост для встраиваемых страниц.
**API:** ядровые WS-функции + наш плагин `local_corplms` (external_api) под то, чего нет в ядре
(агрегаты дашборда, каталог, список сертификатов, оргструктура).

## Фазовый роадмап (вариант C)
- **Ф0. Каркас — ✅ СДЕЛАНО:** Next.js 16/React 19/Tailwind v4 в `frontend/` (compose-сервис), дизайн-система (индиго `#4F46E5`, Manrope+Source Sans 3 с кириллицей), серверный WS-клиент (`node:http`, Host-заголовок), экран «Моё обучение» рендерит **реальный курс** из Moodle через `core_enrol_get_users_courses`. WS включён скриптом `scripts/ws_bootstrap.php` (+ демо-студент demo1, демо-курсы). Открывается на `:3000` (браузер Windows).
- **Ф1. Read-журналы ученика:** логин, дашборд «Моё обучение», каталог/библиотека, обзор курса, профиль, сертификаты — на существующих WS.
- **Ф2. Интеракции:** запись/самозапись, отметка прохождения, просмотр материалов, сдача задания (загрузка файла через WS), опросы. Тесты — встроенный стилизованный Moodle.
- **Ф3. Дашборды менеджера/куратора:** аналитика и отчёты поверх данных + эндпоинты `local_corplms`.
- **Сложное оставляем в Moodle:** попытки тестов, SCORM, авторинг, админка — themed, встроены/по ссылке.

## Риски (держать в голове)
1. **Пробелы WS API** → часть фич требует своих external functions (наш плагин) или остаётся в Moodle.
2. **Плеер тестов/SCORM** — повторить богатство mod_quiz (типы вопросов, грейдинг) с нуля = очень дорого. Встраиваем Moodle.
3. **Апгрейды Moodle** меняют WS-контракты → регресс-тесты фронта обязательны.
4. **Срок тендера** — фронт не должен блокировать поставку; тендер закрываем вариантом A.
5. **ИБ (п.12.2 ТЗ)** — вторая система = отдельный OWASP-аудит, SSO, хранение токенов.

## Что переиспользуется под другие тендеры
- Фронт брендируется токенами/`.env` (цвет, лого, локали) — перекрас под тендер без кода.
- Плагин `local_corplms` с external functions — общий для всех внедрений.
- Это и есть реальный дифференциатор продукта на рынке РК.

## Решение, которое нужно принять
- **Объём:** A-only (быстро, tender-safe) · A+C-каркас (рекомендую) · сразу B (не советую).
- **Если A+C:** с какого журнала ученика начинаем Ф1 (дашборд «Моё обучение» — логичный первый).
